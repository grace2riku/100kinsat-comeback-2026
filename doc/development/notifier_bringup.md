# 状態通知（スピーカ D09 + 内蔵LED×4）立ち上げ・実機確認手順（Issue #15）

ホストテストで確認できるのは「**HW非依存ロジック**」だけ（gotchas A）。具体的には時間経過に対する
出力系列（周波数・LEDマスク）と再生状態の遷移（`test/core/test_notifier.cpp`：`Notifier` /
`patternFor`、1回再生・繰り返し・境界値・異常系）。次の項目は実機でしか確認できない:

- **スピーカ（UGCM0603APE, D09）が実際に鳴る**（tone() の発音・音量・音程の聞こえ方）
- **内蔵LED×4 が点滅する**（視認性）
- **3パターンが聞き分け・見分けられる**（起動/ゴール/エラーの性格の違い）
- **ゴール通知が停止の客観的提示に足るか**（細則 §1「音や光で客観的に分かるのが望ましい」）

本書はその確認手順と記録テンプレート。確認結果を埋めてコミットし、Issue #15 の HW依存 DoD を
満たしたうえでマージする（HWモジュールのマージゲート: 実機確認まで Issue をクローズしない）。

対象コマンド: シェル（`src/shell`）の `notify <boot|goal|error> [sec] | stop`
パターン再生ロジックは `src/lib/core/notifier`、発音・点灯は `.ino` 内の tone()/digitalWrite
（HAL 薄呼び出し。出力は変化時のみ適用）。

---

## 0. ⚠️ 注意・前提

- **notify は音と光のみで走行系を駆動しない**ため安全（gotchas B5）。再生時間は有限
  （goal/error は既定5秒・上限30秒）で、**非改行キーで即中断**できる（gotchas B10）。
- **notify は終了時に全 LED を消灯する**（消音・消灯の終了保証）。`led` コマンドで点けた LED も
  消えるが安全側の副作用（LED は共用リソース）。
- **`[sec]` は goal/error のみ**（boot は1回再生で完了まで鳴るため秒指定は受け付けない）。
- **パターンの実体（音程・点滅）は暫定**（`notifier.cpp` に一元定義）。実機の聞こえ方で調整して
  よいが、3パターンの「性格の違い」（起動=上昇音階1回 / ゴール=ゆっくり断続 / エラー=速い高低
  交互）は保つこと（テストは相互差分のみ検証するため音色調整でテストは壊れない）。
- **実飛行では #17 が結線する**: 状態遷移時に `notifier::Notifier::start(Notice)` を呼び、制御
  ループから `update(dtMs)` で回す（ノンブロッキング）。ゴール通知は stop() まで無限に繰り返す。
- スピーカの実体は BZ1（UGCM0603APE）。鳴らない場合は D09 配線・シルク「＋」の極性
  （hardware.md 組立）を確認する。

## 1. 準備

1. シェルをビルド＆書き込み:
   ```bash
   tools/build.sh  src/shell
   tools/upload.sh src/shell
   ```
2. シリアルモニタを開く（115200bps）。プロンプト `100kinsat>` を確認。
3. `help` で `notify` の書式を確認する。

## 2. 確認手順

### 手順A: 各パターンの単体再生

```
notify boot        （起動音: 上昇4音 C5→E5→G5→C6 が1回。LED0→0-1→0-2→全点灯の順増）
notify goal        （ゴール: 880Hz 断続ビープ+全LED点滅を5秒）
notify goal 10     （再生時間を10秒に）
notify error       （エラー: 1200/600Hz 高低交互+LED交互点滅を5秒）
notify stop        （保険: 消音・消灯）
```

観察ポイント:
- boot は**1回だけ**鳴って自動で消音・消灯するか
- goal / error は指定秒数**繰り返し**、終了後に消音・消灯するか
- 再生中に非改行キー（例: スペース）で**即中断**し、消音・消灯するか

### 手順B: 聞き分け・見分け

- goal と error を続けて鳴らし、**音だけで区別できる**か（ゆっくり断続 vs 速い高低交互）
- LED だけ見て区別できるか（全点滅 vs 交互点滅）

### 手順C: ゴール通知の客観性（細則 §1）

- 機体から数 m 離れて `notify goal 30` を再生し、**環境音がある場所でも聞き取れる**か確認する。
  屋外・審査員距離での聴認は #22（屋外試験）で確認する（申し送り）。

### 手順D: モータPWM との同時使用（tone×analogWrite のタイマ競合。gotchas A6）

本モジュールの存在理由は「**走行中の制御ループから**ノンブロッキングで鳴らす」こと（特に Error
通知は走行中に発火し得る）。CXD56 の `tone()` はハードタイマ、モータPWM（`analogWrite` D03/D05）
もタイマ資源を使うため、**競合すると走行中に音が出ない／PWMデューティが乱れる**可能性がある。
ホストテスト（出力系列の検証）では絶対に顕在化しないため、実機で並走を確認する:

シェルはコマンドを直列実行するため**真の並走はシェル単体では確認できない**。ここでは資源の
相互破壊がないことの簡易確認に留め、真の同時実行は #17 で確認する:

1. 機体を浮かせる（車輪が空転できる状態にする）。
2. `notify goal` → 直後に `motor forward 120 1000` → 直後に `notify goal` を連続実行し、
   (a) 通知の後にモータが正常回転すること、(b) モータ駆動の後も通知が正常に鳴ることを確認する
   （tone とモータ PWM がタイマ資源を相互に壊して残さないことの確認）。
3. **真の同時（音を鳴らしながら PWM 駆動）の確認は #17 の結合実装で必ず行う**（走行中に
   Error/Goal を start して音・回転の両方を観察）。#17 の DoD に追記済み。

## 3. 実機確認チェックリスト（マージゲート）

> HWモジュールのマージゲート: 下記を実機で確認し日付を記録するまで Issue #15 をクローズしない。

- [x] `notify boot` で起動音が1回鳴り、LED が順に増えて自動で消音・消灯する（手順A）
- [x] `notify goal` で断続ビープ+全LED点滅が指定秒数繰り返し、終了後に消音・消灯する（手順A）
- [x] `notify error` で高低交互の警告音+LED交互点滅が再生される（手順A）
- [x] 再生中の非改行キーで即中断し、消音・消灯する（手順A）
- [x] goal と error が音・光で聞き分け/見分けられる（手順B）
- [x] 数 m 離れてもゴール通知が聞き取れる（手順C。屋外・審査員距離は #22 へ申し送り）
- [x] notify とモータ駆動を交互に実行しても双方が正常動作する（手順D。真の同時実行は #17）

### 実機確認結果（2026-07-19）

- 手順A〜D の全チェック項目を実機（src/shell + スピーカ D09 + 内蔵LED×4）で確認し、全て
  期待どおり動作した。パターン値（音程・点滅）の調整は不要（`notifier.cpp` の暫定値のまま採用）。
- 手順D（notify とモータ駆動の交互実行）で tone×モータPWM のタイマ資源が相互に壊れないことを
  確認。**真の同時実行（走行中に鳴らす）は #17 の結合実機確認で行う**（DoD 追記済み。gotchas A6）。
- 屋外・審査員距離でのゴール通知の聴認は #22 の DoD へ申し送り済み。

## 4. 申し送り（#17 ステートマシン / #22 屋外試験）

- **#17（ミッションステートマシン）**: 状態遷移→通知の結線。起動完了で Boot、ゴール確定で
  Goal（stop まで継続＝停止の客観的提示）、異常検知で Error を `start()` し、制御ループから
  `update(dtMs)` を回して `toneHz()/ledMask()` を実 HW へ変化時のみ適用する。
- **#22（屋外試験）**: ゴール通知音が屋外・審査員距離で聴認できるか確認（音量が不足するなら
  音程・パターン長の調整、または旗等の視覚提示の併用を検討）。
