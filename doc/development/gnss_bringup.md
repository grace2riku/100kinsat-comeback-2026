# GNSS立ち上げ・実機検証手順（Spresense内蔵 / Issue #10）

ホストテストで確認できるのは「**HW非依存ロジック**」だけ（gotchas A）。具体的には目標地点との距離・方位計算（`test/core/test_geo.cpp`）と、測位スナップショットから「走行に使える位置か」を判定する FIX/HDOP ロジック（`test/core/test_gnss.cpp`：`hasPositionFix` / `isUsableForNavigation`）。次の項目は実機でしか確認できない:

- **GNSS の初期化・測位開始**（`begin`/`start(COLD_START)` が実機で成功するか）
- **屋外で実際に FIX が取れること**（衛星捕捉 → `fixMode` が 1→2→3 へ収束）
- **緯度経度の物理的な妥当性**（既知地点と一致するか）
- **HDOP が走行可しきい値（既定 5.0）まで下がること**（精度が実用域に入るか）

本書はその計測手順と記録テンプレートを兼ねる。屋外計測の結果を埋めてコミットし、Issue #10 のHW依存DoDを満たしたうえでマージする（HWモジュールのマージゲート: 実機確認まで Issue をクローズしない）。

対象コマンド: シェル（`src/shell`）の `gnss [init|mon [n]]`
測位源: Spresense **内蔵 GNSS**（`GNSS.h`。外付け配線なし）。判定ロジックは `src/lib/core/gnss_fix`、測位取得は `src/lib/hal` の `SpresenseGnss` ラッパが担う。hardware.md / software.md §5.5 参照。

## 0. ⚠️ 注意

- **必ず屋外（見通しの良い場所）で計測する。** 屋内は受信不可または精度劣化する（software.md §5.5）。天空視界（空が広く見える場所）とアンテナ接続を確保する。
- **FIX には時間がかかる。** `gnss init` は `COLD_START`（受信履歴なし）なので、最初の FIX まで**屋外でも数十秒〜数分**かかる。すぐに座標が出なくても異常ではない。
- **自宅等で計測した座標の SNS 公開は避ける**（software.md §5.5）。緯度経度から所在地が特定されるため、共有時はマスクする。
- `gnss` / `gnss mon` は**読み取り専用で HW を駆動しない**ため、`delay` でブロックしても安全（暴走しない。gotchas B5 の駆動系コマンドとは別扱い）。`mon` 中は**非改行キー入力で即中断**できる（CR/LF はコマンド確定の残留改行なので中断トリガーにしない。gotchas B10）。

## 1. 準備

1. シェルをビルド＆書き込み:
   ```bash
   tools/build.sh  src/shell
   tools/upload.sh src/shell
   ```
2. シリアルモニタを開く（115200bps）。**モニタを開いてからボードをリセット**するとバナーとプロンプト `100kinsat>` が見える（A2: `while(!Serial)` で待たない運用）。GNSS は**起動時には初期化しない**（`gnss init` の案内が出る）。
3. **本体を屋外の天空視界が開けた場所**へ持ち出す（窓際より屋外。ビル谷間・樹冠下は避ける）。
4. `help` で `gnss` の書式を確認する。

> なぜ起動時に初期化しないか: GNSS の `begin`/`start` は時間がかかり、未配置（屋内）で起動するたびに待たされると検証効率が落ちる。初期化は `gnss init` のオンデマンドにし、屋外に出てから測位を開始する運用にしている。

## 2. 手順A: 初期化・測位開始（`gnss init`）

屋外で `gnss init` を実行すると次のどちらかが出る:

- `gnss: 初期化OK（COLD_START。FIX まで屋外で数十秒〜数分）`
- `gnss: 初期化失敗（GNSS begin/start エラー。電源・アンテナを確認）` → 電源・アンテナ接続を確認し、`gnss init` で再試行。

未初期化のまま `gnss` / `gnss mon` を実行すると `gnss: 未初期化。'gnss init' で初期化してください` が出る（無効値で先に進まない）。

### 記録

| 状況 | 期待 | 実際 | 判定 |
|---|---|---|---|
| 起動直後 | バナーとプロンプトが出る（起動はハングしない） | プロンプト `100kinsat>` 表示・操作可能 | ✅ |
| `gnss init`（屋外） | 「初期化OK（COLD_START…）」 | 「gnss: 初期化OK（COLD_START。FIX まで屋外で数十秒〜数分）」 | ✅ |
| 未初期化のまま `gnss` | 「未初期化。'gnss init' で…」案内（無効値で誤動作しない） | 未確認（要追試） | ⏳ |

## 3. 手順B: FIX 収束の観察（`gnss mon`）

初期化後、`gnss mon` で測位を連続表示し、衛星捕捉から FIX 収束までを観察する。GNSS は約 1Hz 更新で、1サンプルあたり最大2秒待つ（更新を1つ待って表示）。

操作例（既定20回。回数を増やすなら `gnss mon 60`。上限120）:
```
gnss mon 60
```

各行は `[i/count]` の進捗付きで次の形で出る:
```
[ 1/60] sat= 4 fixMode=1  -> No Position（測位不能：屋外・天空視界を確保）
[ 8/60] sat= 7 fixMode=2 pos=30.123456,130.123456 hdop=3.20 vel=0.10  -> FIXあり/精度不足（HDOP が高い or 0）
[20/60] sat=10 fixMode=3 pos=30.123456,130.123456 hdop=0.90 vel=0.05  -> 走行可（FIX・精度良好）
```

観察ポイント（時間とともに次のように収束していく）:
- **`sat`（見えている衛星数）が増えていく**（捕捉が進む）。
- **`fixMode` が 1(Invalid)→2(2D FIX)→3(3D FIX)** と進む。
- **`pos`（緯度経度）が安定**し、暴れがなくなる。
- **`hdop` が下がっていく**（精度が上がる。小さいほど高精度）。

中断したいときは Enter 以外のキーを1打する（`gnss mon: 中断しました` が出て止まる）。GNSS は約1Hz 更新のため、2秒窓で新サンプルを拾えなかった周期には `gnss: この周期は更新なし（約1Hz。未FIXが続く場合は屋外で天空視界を確保し再試行）` が出る。FIX 済みでも更新位相のずれで時々出るのは正常（衛星喪失ではない）。これが**連続して**続く場合のみ天空視界を疑う。

### 記録

計測日: 2026-06-28 ／ `gnss mon 60` ／ 場所: 自宅付近（座標は下位桁マスク、真値は非公開で別途記録）

| 経過 | sat | fixMode | hdop | pos の安定 | 備考 |
|---|---|---|---|---|---|
| `gnss mon` 開始直後 | 8 | 1 | — | No Position | `[1/60]`〜`[22/60]` は No Position（`pos` 非表示）。sat は 8→10 と増加 |
| 初 FIX（2D）到達 | — | (2) | — | — | **2D FIX は観測されず**。`[23]` のタイムアウト直後 `[24]` で 1→3 へ直接遷移（COLD からの収束が速かった） |
| 3D FIX 到達 | 9 | 3 | 4.51 | 出始め | `[24/60]` で初 FIX。`pos=35.76xx,139.48xx`（マスク） |
| 走行可（精度良好）到達 | 9 | 3 | 4.51 | 安定 | `[24/60]` の初 FIX で即 `-> 走行可（FIX・精度良好）`（HDOP≤5.0）。以後 `hdop` は 2.17 まで収束、`sat` は 12 まで増加 |

**初 FIX（fixMode≥2）までに要した時間**: 約 25 秒（※ログにタイムスタンプがないため、約1Hz 更新としてサンプル数からの概算。`[24]` 番目で到達）

> 未観測の状態: 中間状態 `-> FIXあり/精度不足（HDOP高い/0）` は本計測では出ていない（初 FIX 時点で既に HDOP≤5.0 だったため）。判定ロジック自体は `test/core/test_gnss.cpp` でホストテスト済みだが、実機での当該表示は §7 の DoD として別途観測したい（HDOP が高く出る悪条件か、しきい値を一時的に厳しくして再現）。

## 4. 手順C: 既知地点での座標妥当性確認

3D FIX・走行可になったら、**位置の分かっている地点**（建物の角、目印など）に本体を置き、`gnss` で1回表示して緯度経度の妥当性を確認する。

```
gnss
```

- 表示された `pos=緯度,経度` を地図サービス（既知地点の座標）と照合する。
- CanSat の数百m走行スケールでは、**数m〜十数m程度のずれは正常**（HDOP 由来の誤差）。桁が違う／海上や別の国を指す場合は座標の取り違え・単位ミスを疑う。

## 5. 出力の読み方（3状態）

`print_gnss` は FIX/HDOP 判定（core/gnss_fix のホストテスト済ロジック）に基づき末尾を3状態で出し分ける:

| 末尾表示 | 条件（core/gnss_fix） | 意味 | 走行に使えるか |
|---|---|---|---|
| `-> No Position（測位不能：屋外・天空視界を確保）` | `hasPositionFix==false`（`posDataExist=0` / `fixMode` が 1/Invalid・化け値 / 座標範囲外 / 座標が `(0,0)`） | 位置が測位できていない。`pos` 自体を表示しない | ✕（座標が不定） |
| `-> FIXあり/精度不足（HDOP が高い or 0）` | `hasPositionFix==true` だが `isUsableForNavigation==false`（HDOP が `maxHdop` 超 or `hdop<=0`） | FIX はあるが精度が実用域に届かない | ✕（精度不足） |
| `-> 走行可（FIX・精度良好）` | `hasPositionFix==true` かつ `isUsableForNavigation==true`（HDOP が `(0, 5.0]`） | FIX かつ精度良好。ナビに使える | ◯ |

HDOP（水平精度劣化指数）の読み方:
- **小さいほど高精度**（理想 ≤1、良好 ≤2、中 ≤5）。
- 走行可の境界は既定 `maxHdop = 5.0`（`src/lib/core/gnss_fix.h` の `kDefaultMaxHdop`。実機の収束を見て調整可）。
- **`hdop = 0` は「未取得/無効」の典型値**で、高精度（0=完璧）と誤認しないよう**無効として弾く**（判定は `hdop > 0 && hdop <= maxHdop`）。負値・NaN も同様に範囲外で弾かれる。
- `fixMode` は仕様上 1/2/3 のみ。化けた値（4以上）を FIX と誤認しないよう範囲一致で判定している。

## 6. トラブルシュート

| 症状 | 確認・対処 |
|---|---|
| `No Position` が続く（`fixMode=1` のまま） | **屋外・天空視界・アンテナ接続**を確認。窓際や樹冠下・ビル谷間は受信が弱い。開けた場所へ移動して数分待つ |
| `この周期は更新なし` が単発で出る | 正常。約1Hz 更新を2秒窓で拾えなかっただけで、FIX 済みでも時々出る（衛星喪失ではない） |
| `この周期は更新なし` が連続する／未FIXのまま続く | 衛星未捕捉を疑う。屋外で天空視界を確保し再試行。初回 COLD_START は時間がかかる |
| FIX まで非常に時間がかかる | `COLD_START` は受信履歴なしから測位するため遅い（測位速度 HOT > WARM > COLD）。**短時間に再起動を繰り返す運用**なら、将来 `WARM_START`/`HOT_START` の採用も検討（software.md §5.5）。本検証は COLD_START で行う |
| `gnss: 初期化失敗（…）` | 電源・アンテナ接続を確認して `gnss init` を再試行 |
| `pos` が出るが `hdop` が高い/0 | 精度不足。空の開けた場所で `gnss mon` を続け、`hdop` が下がる（衛星が増える）のを待つ |

## 7. DoD 対応（Issue #10）

本手順は**実機の物理的な測位取得**を確認するためのもの。位置づけを明確にしておく:

- **HW非依存ロジックはホストテスト済**: 目標地点との距離・方位計算（`geo`）と、FIX/HDOP 判定（`core/gnss_fix`：`hasPositionFix` / `isUsableForNavigation`）は `test/core/` で検証済み。座標が妥当なら正しく走行可否を返すことは構成的に担保されている。
- **実機で確認すべき DoD**: 屋外で実際に FIX を取得し、§5 の3状態表示が定義どおり出ること。特に
  - **測位不能（`posDataExist=0`）** のとき `No Position` が出て `pos` を表示しないこと、
  - **低精度（HDOP が高い/0）** のとき `FIXあり/精度不足` が出ること、
  - **FIX かつ HDOP 良好** のとき `走行可（FIX・精度良好）` が出ること、
  を実機の収束過程（手順B）で一通り観測する。
