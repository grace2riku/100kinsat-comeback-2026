# 制御履歴ログ（SD: 制御量＋操作量）実機 bring-up 手順

Issue #14。**本種目で順位がつくための必須要件**（細則§5「制御履歴とは」/ 大会要領 種目5）。
制御量（観測値＝位置・速度・方位）と操作量（モータ指令）を時系列で SD に CSV 保存する。
これが無いと「偶然ゴールに向かった」のと区別できず順位がつかない。車輪破損・スタックで指令
通り動かなくても、操作量の記録があれば制御指令までは正常として**部分点**評価に使える。

- CSV 整形・欠損サニタイズ・周期 flush は HW非依存 core `src/lib/core/datalog.{h,cpp}`
  （ホストテスト済 `test/core/test_datalog.cpp`。件数は `ctest`/`grep -c TEST_CASE` を正とする）。
- SD(SDHCI/FAT32) 追記・連番ファイル名・同時1ファイル制約は `src/lib/hal/sd_log_sink.h`。
- 実機結線は `src/shell` の `log` コマンド。

---

## CSV フォーマット（12列）

```
t_ms,state,fix,lat_deg,lon_deg,vel_mps,course_deg,heading_deg,dist_m,bearing_deg,motor_l,motor_r
```

| 列 | 意味 | 区分 | 書式 | 欠損時 |
|---|---|---|---|---|
| `t_ms` | 起動からの経過 [ms]（millis 相当） | 時刻 | 整数 | — |
| `state` | ミッション状態コード（下表） | 状態 | 整数 | — |
| `fix` | GNSS 位置が有効か（1/0） | — | 0/1 | — |
| `lat_deg` | 緯度 [deg]（観測/推定） | 制御量 | 小数7桁 | fix=0 で空 |
| `lon_deg` | 経度 [deg] | 制御量 | 小数7桁 | fix=0 で空 |
| `vel_mps` | 対地速度 [m/s]（GNSS） | 制御量 | 小数2桁 | fix=0 で空 |
| `course_deg` | 進行方位 [deg]（GNSS/観測） | 制御量 | 小数1桁 | fix=0 で空 |
| `heading_deg` | 姿勢方位 [deg]（9軸/制御用） | 制御量 | 小数1桁 | 非有限で空 |
| `dist_m` | 目標までの距離 [m]（geo） | 制御量 | 小数2桁 | fix=0 で空 |
| `bearing_deg` | 目標への方位 [deg]（geo） | 制御量 | 小数1桁 | fix=0 で空 |
| `motor_l` | 左モータ指令（-255..255, 負=後進） | **操作量** | 整数 | — |
| `motor_r` | 右モータ指令（-255..255, 負=後進） | **操作量** | 整数 | — |

- **欠損表現**: 数値が非有限（NaN/Inf）、または位置由来（lat/lon/vel/course/dist/bearing）で
  `fix=0` のときは**空フィールド**（`,,`）にする。未測位の残骸を履歴へ流さない（gotchas B12）。
  後解析（pandas 等）では空 = 欠損（NaN）として扱える。
- `heading_deg` は 9軸姿勢由来なので GNSS FIX に依存せず、有限なら出力する。

### state 対応表（暫定・#17 で確定）

`state` 列には #17 ミッションステートマシンの `MissionState` を uint8 で格納する想定。
現時点の暫定コード（**#17 実装時に確定・本表を更新**する）:

| code | 状態 | 意味 |
|---|---|---|
| 0 | INIT | 起動・初期化・放出待ち |
| 1 | DESCENT | 降下中（パラシュート） |
| 2 | DRIVE | 自律走行中 |
| 3 | GOAL | ゴール停止 |
| 4 | ERROR | 異常 |

> core `datalog` は `state` の値を**透過するだけ**（意味は #17 が定義）。上表は後解析の凡例。

## ファイル名規約

- 保存先: `log/flightNNN.csv`（`NNN` = ゼロ埋め3桁の連番。`kLogDir="log/"`）。
- `hal::SdLogSink` が **既存を上書きしない空き連番**を採番する（`SD.exists` で走査）。
  電源再投入ごとに新ファイルになるので、過去フライトのログが消えない。
- **同時に開くファイルは1個**の制約（software.md §5.7）を守る（本 sink は1ファイルのみ保持）。

## flush ポリシー

- `FlightLogger` は **既定 10 レコードごとに `flush()`**（`kDefaultFlushEveryN`）してメディアへ確定する。
- 電源断で失われるのは直近 flush 以降（最大 `flushEveryN-1` 件）に限られる。走行の記録間隔と
  電池・電源断リスクを見て間隔は調整する（#17 で確定）。

## 0. 前提

- microSD（**FAT32 フォーマット必須**）をメインボードのソケットへ挿入。
- `tools/upload.sh src/shell <port>` で書き込み、`tools/monitor.sh <port>`（115200bps）で接続。

## 1. 手順A: SD へ記録できることの確認

```
log            （既定5件のダミー制御履歴を記録）
log 20         （20件記録）
```

観察ポイント:
- `log: 記録先 log/flightNNN.csv` が表示され、`log: N 件を記録しました` で完了する。
- SD が無い/FAT32 でないときは `SD を開けません` を返す（誤動作せず失敗を明示）。

## 2. 手順B: CSV の中身を検証（SD を PC で開く）

ボードから SD を抜いて PC で `log/flightNNN.csv` を開く（または別途 `read` 系で吸い出す）。

- [ ] 1行目が**ヘッダ**（上記12列）。
- [ ] `fix=1` の行は lat/lon/vel/course/dist/bearing が**桁数どおり**に入る。
- [ ] `fix=0` の行は位置由来が**空フィールド**（`,,`）、`heading`・`motor`・`state` は出る。
- [ ] `motor_l` に**負値（後進）**が正しく入る（例 `-150`）。
- [ ] 表計算/pandas で読め、空セルが欠損として扱える（後解析しやすさ）。

## 3. 手順C: 連番・flush・電源断の確認

- [ ] `log` を複数回叩くと `flight000, flight001, ...` と**連番が増え**、過去ファイルが残る。
- [ ] 記録の途中で電源を切っても、直近 `flush` までの行は残っている（最大 `flushEveryN-1` 件の損失）。
  - **周期 flush を観察するには `log 20` など n≥`flushEveryN`(既定10) で実行**する。既定の `log`
    （5件）では周期 flush が発火せず、末尾の明示 flush のみになり周期挙動を確認できない。

---

## 4. 申し送り（#17 ミッションステートマシン / #18 自律走行）

- **実センサ値の結合は #17/#18**。本コマンドは単体確認用にダミー `LogRecord` を書く。#17 で
  「毎制御周期に GNSS/9軸/geo/モータ指令を `LogRecord` に詰めて `FlightLogger.log()` する」結線を行う。
- `state` の**確定コードを本書の対応表に反映**する（#17 の `MissionState` 実装時）。
- **記録開始/終了のタイミング**: 大会要領は「放出後〜動作停止まで」の履歴を要求。#17 で放出検知（#11）
  でロギング開始、ゴール停止（#15 で客観提示）で終了、という区間管理を実装する。
- **flush 間隔**（既定10）と**モータ指令の取得元**（motor モジュールに現在指令の getter を足すか、
  制御側が値を保持して渡すか）を #18 で確定する。
- **失敗種別の区別**: 現状 `FlightLogger.log()` は「単発の整形/append 失敗」も「SD 消失」も同じ
  `false` を返す。デモ（本コマンド）は最初の失敗で中断してよいが、飛行ロギングにこの方針を
  そのまま持ち込むと**一時的失敗1回で以降の制御履歴が全欠落**する（順位判定の必須要件が飛行後半で
  消える）。#18 では「単発失敗は継続・SD 消失は停止」を区別する（失敗種別の返し分け or 連続失敗
  カウンタでの停止判断）を実装する。
