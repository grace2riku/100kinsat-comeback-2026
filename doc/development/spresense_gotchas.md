# Spresense 実装の落とし穴集（実装前チェックリスト）

実装・レビュー前に必ず目を通すこと。ここに載っているのは **過去に Codex レビューで実際に指摘された不具合** を、再発防止のためルール化したもの。新しい指摘を受けたら、その都度この表に1行追加して「資産」に変える（同じクラスの不具合を二度とPRに到達させない）。

各ルールは「**症状 → 原因 → ルール**」で書く。実装中は該当する作業をするたびに対応行を確認し、レビュー時はこの表をチェックリストとして使う。

---

## A. ターゲット ≠ ホスト の差異（最重要・CIが緑でも実機で壊れる）

ホストPC（64bit Linux/macOS）でユニットテストが通っても、Spresense（32bit ARM）では挙動が変わる項目。**ホストテストの緑は実機の正しさを保証しない。** 該当箇所は実機（NT-Shell）でも必ず確認する。

| # | 症状 | 原因 | ルール |
|---|---|---|---|
| A1 | 整数引数のオーバーフローが検出されず、巻き戻った値で誤動作 | Spresense では `long` も **32bit**。ホスト(`long`=64bit)では桁あふれが顕在化しない | 整数パースは `int` で直接累積し、各桁で `INT_MAX`/`INT_MIN` 超過を検査して**拒否**する。`long` 経由の検査に頼らない。負値は `INT_MIN` を正しく扱うため下向きに累積する |
| A2 | 起動バナーが出ず、シリアルモニタを開いても先頭ログが見えない | Spresense の `Serial` は USB-UART(CP210x)。`HardwareSerial::operator bool()` は `_fd >= 0` を見るだけで、モニタ接続を検知できない。`while(!Serial)` は**待たない** | `while(!Serial)` でモニタ接続を待つコードを書かない。代わりに `delay()` か、モニタを開いた**後にボードをリセット**する運用にする。手順書にもリセット手順を明記 |
| A3 | （一般則）`size_t`/ポインタ幅・`time_t`・浮動小数の差 | ホストとビット幅・実装が異なる | 幅依存の演算は固定幅型（`int32_t` 等）を使う。ホストテストで通っても「ターゲットでの幅」を意識する |
| A4 | 角度正規化の結果が `-0.0` になり、`0 <= θ < 360` を不変条件とする後段（表示・比較・しきい値）で `-0` 表示や符号依存の判定がぶれる | `fmod` は被除数の符号を残し、`fmod(-360,360)` 等で `-0.0` を生む。`-0.0 < 0.0` は false なので「正へ折り返す」補正をすり抜ける | 角度・距離など「非負を不変条件にする量」は、折り返し後に**正のゼロへ正規化**する（`if (r == 0.0) r = 0.0;`）。テストは `Approx` だけでなく `std::signbit` で符号まで検査する。`src/lib/core/compass.cpp` の `normalizeHeading` が実例 |

> **運用ルール:** ホスト非依存ロジックでも、整数範囲・ビット幅・シリアルが絡む箇所は「ホストテスト緑 = 完了」ではなく、**NT-Shell で実機確認**するまで完了としない。

## B. 入力検証・境界・異常系（CLAUDE.md の最重視領域）

シリアルコマンドや外部入力は「人間が手で打つ」前提で、**不正入力でハードを誤駆動させない**こと。

| # | 症状 | 原因 | ルール |
|---|---|---|---|
| B1 | `led foo on` の `foo` が 0 と解釈され LED0 を誤駆動 | `atoi` は非数値を 0 として返し、失敗を区別できない | コマンド引数の数値化に `atoi`/`atol` を**使わない**。符号+数字のみを許可し、余分な文字・空文字を**失敗**にする厳密パーサ（`src/lib/core/cli` の `parseInt`）を使う |
| B2 | 範囲外の値（過大なLED番号・beep長）が通る | 範囲チェック漏れ・パース成功扱い | パース成功後に**必ず範囲（最小・最大）を検査**し、範囲外はエラーにして**HWを駆動しない** |
| B3 | LF-only モニタで Enter が効かず `help` 等が実行されない | 改行コードの片対応（CR(0x0d)だけ、または LF(0x0a)だけ） | シリアル入力は **CR / LF / CRLF いずれも**確定として扱う。CRLF の LF 二重確定にも注意（直前が CR なら読み飛ばす） |
| B4 | バイト未読のままパーサに未初期化値が渡りゴミが混入 | `read` の戻り値（読めたバイト数）を無視 | `read` 系は**戻り値を確認**し、`n <= 0` のときはパーサ／処理を呼ばない |
| B5 | アクチュエータ（モータ等）を一定時間駆動するコマンドで、駆動時間に上限が無く `delay()` 一括で固まり、暴走時に緊急停止が届かない | 駆動時間の上限チェック漏れ＋ブロッキング待ち | HWを時間駆動するコマンドは**駆動時間に上限**を設け範囲外を拒否する。飛行制御など停止応答性が要る箇所では `delay()` 一括を避け、`millis()` ベースで分割しつつ停止入力をポーリングする |
| B6 | 余剰引数（`motor forward 120 500 extra`）を黙って無視して HW を駆動する | 引数数の**下限のみ**検査し上限（余剰）を見ていない | HWを駆動するコマンドは**期待トークン数の上下限**を検査し、余剰引数は駆動前に拒否する（既存 led/beep は厳密 arity 検査）。停止など安全側コマンドは寛容でよい |
| B7 | センサの状態値（校正レベル 0-3 等）を `>=` で閾値比較し、I2C 化け・未初期化で混入した異常値（4 以上）を「正常/完了」と誤判定して HW を駆動 | 値域の**上限検査漏れ**（`>=3` は化け値も通す）＋ C 宣言 `uint8_t a,b,c,d=0;` の**部分初期化の罠**（先頭3つは未初期化） | センサ由来の整数状態は使用前に**下限・上限の両方**を検査する。「完了」判定は `>=` ではなく**有効上限との一致（`==3`）**で安全側に弾く。複数変数のゼロ初期化は**1つずつ**確認する（`uint8_t a=0,b=0,c=0,d=0;`）。`src/lib/core/compass.cpp` の `isFullyCalibrated`/`isHeadingReady` が実例 |

## C. ファイル間・スクリプト間の整合性

「片方だけ直す」を防ぐ。対になっているものは常に**両方**更新する。

| # | 症状 | 原因 | ルール |
|---|---|---|---|
| C1 | `tools/upload.sh` で書き込み時にライブラリ未解決でビルド失敗 | `tools/build.sh` に `--libraries` を足したが `upload.sh` に反映漏れ | `build.sh` と `upload.sh` は同じ `arduino-cli compile` オプション（`--libraries` 等）を**ミラーする**。片方を変えたら必ずもう片方も |
| C2 | （一般則）ヘッダ宣言と実装、ドキュメントとコードの不一致 | 対の更新漏れ | シグネチャ・定数・手順を変えたら、宣言／実装／ドキュメント／テストの**4点セット**を同時に更新 |
| C3 | スクリプト(precheck/CI)と運用ドキュメントで、同一ツール（codex 等）の起動主体・自動/手動・前提が食い違う | 仕組みを再設計したのにドキュメント側の役割表・手順名を更新し忘れる | スクリプトの**ステップ名・前提（誰が・自動か手動か）**と `development_workflow.md` の §5/§7 役割表は**対で更新**する。`code-reviewer` サブエージェント自身のレビューで検出（自己ドッグフーディング） |
| C4 | ローカルゲート(`precheck.sh`)が CI の一部ジョブを既定で省き、ローカル緑でも PR で落ちる（例: 引数なしだと Arduino ビルドをスキップ） | 「CI先取り」と謳いつつ CI の全ジョブをミラーしていなかった | `precheck.sh` は **CI(.github/workflows/ci.yml) の全ジョブ**（host-test/lint/arduino-build）を既定で先取りする。CI 側のビルド対象（`blink_led`/`flight`/`shell`）を変えたら precheck の既定リストも対で更新する |

## D. ドキュメントと実体の乖離

手順書は「実際にその通り操作して動く」状態を保つ。

| # | 症状 | 原因 | ルール |
|---|---|---|---|
| D1 | 手順に存在しないメニュー名（例: "Program Bootloader"）を記載 | 実機UIを確認せず推測で書いた | UI 操作手順は**実際のメニュー名・順序**を確認して書く。公式セットアップガイドと突き合わせる |
| D2 | 手順の順序が誤りで失敗（ポート選択がブートローダ書込より後 等） | 依存関係を無視した順序 | 「この操作はこの前提が必要」を意識し、**依存順**に並べる。Firmware Updater は選択中ポートに対し動く→ポート選択を先に |
| D3 | 「`.claude/settings.local.json` はgitignore対象」と書いたが、無視していたのは**個人のグローバル** `~/.gitignore_global` で**リポジトリの `.gitignore` には未記載**。他メンバーは追跡対象になり個人設定を誤コミットしうる | `git check-ignore` がグローバル設定で真を返し「リポジトリで担保済み」と錯覚した | **チーム共有の無視規約はリポジトリの `.gitignore` に明記**する。確認は `git -c core.excludesfile=/dev/null check-ignore <path>`（グローバルを無効化）で行い、リポジトリ単体で無視されるか検証する |
| D4 | Write/生成ツール由来のテンプレート閉じタグ（例: 末尾の閉じタグ）がドキュメントに混入したままコミットされ、手順の正本に正体不明のタグが残る | 生成物を目視確認せずコミットした | `tools/precheck.sh` の **leaked-marker チェック**で機械的に検出（`git grep` で固有マーカーを走査。本チェッカー自身は除外）。新規ファイルは Write 後に末尾を確認する |
| D5 | 値/状態/実装を変えた（例: スタブ→実装、ピン割当の入替）のに、別ファイルの説明が旧のまま残る（ヘッダコメント・別手順書の状態表・README・テストのコメント）。Codexに何度も同クラスを指摘される | 変更箇所だけ直し、**散在する参照を洗い出していない** | 変更したら **`grep -rn '<機能名/定数>' src/ doc/ test/ README.md` で全参照を走査**し、表記（スタブ/実動作・左右割当・型）を一括更新する（C2の徹底）。「コマンド表だけ直してヘッダ/別手順書を忘れる」が典型 |

## E. ツールチェイン・バージョン前提

「自分の環境で動く」≠「宣言した最小バージョンで動く」。

| # | 症状 | 原因 | ルール |
|---|---|---|---|
| E1 | CMake 3.16〜3.19 で `ctest --test-dir` が無く失敗 | `--test-dir` は CTest **3.20+**。`cmake_minimum_required(3.16)` と矛盾 | 宣言した最小バージョンで動く機能だけ使う。新しい機能が必要なら最小バージョンを上げるか、互換手段（サブシェルで `cd` 等）を使う |
| E2 | `compile_commands.json` の参照先が消えて clangd が機能しない | 一時ビルドパスを生成直後に `rm -rf` し、DB が存在しないファイルを指す | 補完DBが参照するビルド成果物は**永続化**する（`.arduino-build/`）。生成後に「DBの全エントリの参照先が実在するか」を検証 |

## 使い方（実装／レビューの両局面）

1. **実装前**: これから触る領域（整数パース・シリアル・スクリプト・手順書・CMake）に対応する行を読む。
2. **実装後・PR前**: `doc/development/development_workflow.md` のセルフレビューチェックリストと併せて、この表を上から照合する。
3. **新しい指摘を受けたら**: 該当クラスタに1行追加し、可能ならホストテスト（再現テスト）も書く。指摘を「資産」に変えるのがこのファイルの目的。
