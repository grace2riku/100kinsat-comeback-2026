# 9軸センサ立ち上げ・実機検証手順（BNO055 / Issue #9）

ホストテスト（`test/core/test_compass.cpp`）で確認できるのは「方位の正規化・最短回頭角・校正完了判定」という**HW非依存ロジック**だけ（gotchas A）。次の項目は実機でしか確認できない:

- **センサの検出・I2C 初期化**（未検出時のエラーハンドリング）
- **キャリブレーションが実際に 0→3 へ収束すること**
- **方位角の物理的な正しさ**（北で 0°付近、時計回りで増加するか／磁気偏角・取付向き）
- **`isHeadingReady`（sys+mag==3）の暫定しきい値が方位精度に十分か**（gyro 未校正時の挙動）

本書はその計測手順と記録テンプレートを兼ねる。計測値を埋めてコミットし、Issue #9 のHW依存DoDを満たしたうえで再導入/マージする（HWモジュールのマージゲート: 実機確認まで Issue をクローズしない）。

対象コマンド: シェル（`src/shell`）の `imu [init|cal|mon [n]]`
結線: BNO055 を I2C で接続（`D14=SDA / D15=SCL`、アドレス `0x28`）。hardware.md / software.md §5.6 参照。

---

## 0. ⚠️ 注意

- 配線（SDA/SCL/電源/GND）を**通電前に**確認する。I2C アドレスは出荷時 `0x28`。ジャンパで `0x29` にした場合は `src/shell/shell.ino` の `static hal::Bno055Compass g_imu;` を `static hal::Bno055Compass g_imu(0x29);` に変えて再ビルドする。
- 方位は地磁気を使うため、**モータ・スピーカ・大電流配線・金属・磁石から離して**校正・計測する（磁気が乱れると方位が大きくずれる）。
- `imu mon` は読み取り専用で HW を駆動しないため、`delay` でブロックしても安全（暴走しない。gotchas B5 の駆動系コマンドとは別扱い）。

## 1. 準備

1. Adafruit ライブラリを導入済みか確認（未導入なら [build_arduino_cli.md §2.5](./build_arduino_cli.md)）:
   ```bash
   arduino-cli lib list | grep -i "Adafruit BNO055"
   ```
2. シェルをビルド＆書き込み:
   ```bash
   tools/build.sh  src/shell
   tools/upload.sh src/shell
   ```
3. シリアルモニタを開く（115200bps）。**モニタを開いてからボードをリセット**するとバナーが見える（A2: `while(!Serial)` で待たない運用）。9軸センサは**起動時には初期化しない**（`imu init` の案内が出る。理由は下記）。
4. `help` で `imu` の書式を確認し、`imu init` で初期化する。

> なぜ起動時に初期化しないか（gotchas B9）: Adafruit の `begin()` はソフトリセット後の CHIP_ID 待ちが無限ループになり得る（一度 ACK 後にブラウンアウト/活線抜けで応答が戻らない場合）。`setup()` で呼ぶとシェル全体が起動前にハングし、無関係なコマンドも使えなくなる。よって初期化は `imu init` のオンデマンドにし、最悪ハングしても影響を1コマンドに閉じている（復帰はボードリセット）。

## 2. 手順A: 検出・初期化（未検出ハンドリング）

`imu init` を実行すると次のどちらかが出る:
- `imu: BNO055 検出・初期化 OK`
- `imu: BNO055 未検出（…）` → 配線・電源・アドレスを確認し、`imu init` で再試行。

### 記録

| 状況 | 期待 | 実際 | 判定 |
|---|---|---|---|
| 起動（センサ未配線でも） | バナーとプロンプトが出る（**起動はハングしない**）。`imu init` 案内表示 | バナーとプロンプトが出た（100kinsat> imu: 9軸センサ(BNO055)は 'imu init' で初期化してください） | ○ |
| 正常配線で `imu init` | 「検出・初期化 OK」 | imu: BNO055 検出・初期化 OKと表示された | ○ |
| SDA か SCL を外して `imu init` | 「未検出」エラー（最初から不在ならハングしない） | BNO055は半田付けされていてSDA・SCLを外すのが困難なので確認対象外とする |  |
| 未初期化のまま `imu`（読み取り） | 「未初期化」案内（無効値で誤動作しない） | BNO055は半田付けされていてSDA・SCLを外すのが困難なので確認対象外とする |  |
| 検出後に SDA か SCL を外して `imu stat` | 「異常」（sys_status≠5）。健全性は **`imu stat` で確認する** | BNO055は半田付けされていてSDA・SCLを外すのが困難なので確認対象外とする |  |

> 重要1: 起動（`setup()`）では `begin()` を呼ばないため、センサ故障で**シェル起動が止まることはない**（gotchas B9）。ただし `imu init` 実行中に「一度 ACK 後リセット応答が戻らない」中途半端な故障が起きると、Adafruit `begin()` 内部の CHIP_ID 待ちで**その1コマンドはハングし得る**（復帰はボードリセット）。これは他社同期APIの制約。
>
> 重要2: 活線抜け時の `heading` は getVector の仕様上 `0.0`（=北）になり**値だけでは故障と真北を区別できない**（gotchas B8）。故障検知は必ず `imu stat`（`sys_status==5` で融合稼働中）で行う。
>
> DoD「センサ未検出時のエラーハンドリング」の確認点: 未検出時に**無効値で先に進まない**こと＋**起動がハングしない**こと。

## 3. 手順B: キャリブレーション収束

平面に置いて起動後、本体をゆっくり**8の字**に回す／各軸まわりに傾けると校正が進む。`imu mon 60` で約30秒ぶん連続表示し、`sys/gyro/accel/mag` が 3 に近づくのを観察する。

操作例:
```
imu mon 60
```

> ⚠️ **accel は8の字では校正できない（仕様。実機検証メモ）**: BNO055 の加速度校正は「静止時の重力ベクトル(1G)を ±X/±Y/±Z の**6方向**から観測する」手順なので、回し続ける8の字（mag/gyro 用）では `accel` が `1` 付近で停滞する。これはバグではない。`accel` を `3` にするには、本体の**6面を1回ずつ真下（机に接地）にして各数秒静止**させる「6姿勢の静止校正」が必要。
>
> 重力(↓1G)を、センサ各軸の＋と－の計6方向から当てる（3軸 × ±＝6姿勢）:
> ```
>         ↑ +Z軸
>         │
>  -X ────●──── +X        3軸 × (＋と－) = 6姿勢
>         │              （+Y=奥 / -Y=手前）
>         ↓ -Z軸
> ```
> 箱の面の名前（基準姿勢）:
> ```
>         ┌──────────┐
>        ╱   天面    ╱│
>       ╱          ╱ │
>      ┌──────────┐  │ 右面
>      │          │  │
>      │   前面    │ ╱
>      │          │╱
>      └──────────┘
>    （奥=後面 / 左横=左面 / 下=底面）
> ```
> 図の読み方（★＝天面の目印。回しても★を追えば向きが分かる。▼＝接地面=重力が当たる面）:
> ```
> 〔正面図〕  天★  ← 上を向く面
>        ┌─────────┐
>      左 │  前面   │ 右   ← 手前=前面 / 奥=後面
>        └─────────┘
>        ▼底面 ▔▔▔机  ← ▼が接地面（重力が当たる面）
> ```
> 机（`▔▔▔`）に対する6姿勢。**接地面▼**を1面ずつ全部変える（①〜④は正面から、⑤⑥は横から見た図）:
> ```
> ① 底面を下（平置き）        ② 天面を下（上下ひっくり返す）
>         天★                       底
>    ┌─────────┐               ┌─────────┐
>  左 │  前面   │ 右          右 │  前面   │ 左
>    └─────────┘               └─────────┘
>    ▼底面 ▔▔▔机              ▼天面★ ▔▔▔机
>
> ③ 右面を下（右に倒す）      ④ 左面を下（左に倒す）
>         左                        右
>    ┌─────────┐               ┌─────────┐
>  底 │  前面   │ 天★       天★ │  前面   │ 底
>    └─────────┘               └─────────┘
>    ▼右面 ▔▔▔机              ▼左面 ▔▔▔机
>
>   〔ここから横から見た図 = 右面が手前〕
> ⑤ 前面を下（前に倒す）      ⑥ 後面を下（後ろに倒す）
>         後                        前
>    ┌─────────┐               ┌─────────┐
>  底 │  右面   │ 天★       天★ │  右面   │ 底
>    └─────────┘               └─────────┘
>    ▼前面 ▔▔▔机              ▼後面 ▔▔▔机
> ```
> ①→②は「ひっくり返す」、①→③④は「左右に倒す」、①→⑤⑥は「前後に倒す」。★（天面）が上→下→右→左…と動くのを追えば同じ箱を回しているのが掴める。順番は問わない。
> 操作のコツ: `imu mon 120` を回しっぱなしにして、**各姿勢で3〜5秒、完全に静止**させながら①〜⑥を一巡する。姿勢の切り替えは衝撃を与えずゆっくり。6面を巡ると `accel` が `1→2→3` と上がり、`-> 完全校正(全要素3)` が出る。`accel` が上がらない面で長めに静止させると埋まる。

### 記録

| 経過 | sys | gyro | accel | mag | 備考 |
|---|---|---|---|---|---|
| 開始直後 | 0 | 3 | 0 | 0 | gyro は静止で即 3 |
| mag が 3 になった時点 | 3 | 3 | 0 | 3 | 8の字で到達。同時に `-> 方位は使用可(暫定)` 点灯 |
| 全要素 3 到達 | 3 | 3 | 3 | 3 | 電源ONのまま2回目 `imu mon` の**6姿勢**で accel が `0→1→3` に入り `-> 完全校正(全要素3)` 表示 |

**全要素 3 まで（完全校正）に要した時間**: 約 70 秒（`imu mon` 0.5s/サンプル換算で2回ぶん。1回目で sys/gyro/mag、2回目の6姿勢で accel）。**電源を切らなければ校正値は保持**され、2回目開始時点で `3/3/1/3` から再開できた。
**`-> 方位は使用可(暫定: sys+mag=3)` が出た時点の gyro/accel**: gyro=3 / accel=0

> 実機メモ（収束挙動）: `sys` は急な動き/磁場変化で一瞬 `3→2→3` と落ちるが正常（融合の総合指標で、すぐ復帰）。accel は8の字では入らず、6姿勢の静止校正で `0→1→3` と進む（仕様。上記の図参照）。
>
> ⚠️ 実機メモ（I2C読み取りエラー = **未解決・要対処**）: 校正中、`ERROR: Failed to read from i2c (errno = 0)` がバースト（2〜5連続）で発生（1回目 約18% / 2回目 約7%）。各エラーで `heading = 0.0 deg` が出る（gotchas B8 の「失敗を値で真北と区別できない」実例）。**本体を静置した区間ではエラーが出ず、動かしている区間に集中**（movement-correlated）。
> 結線の事実（hardware.md）: Spresense⇔BNO055 は**PCBパターン直結**（コネクタ・ケーブル無し）、BNO055は秋月モジュールを **U10 にハンダ付け**、**I2Cは BNO055 単独**（バス容量・競合・コネクタ起因は除外）。よって movement-correlated を説明する仮説は2つ:
> 1. **BNO055 のクロックストレッチ vs Spresense I2C マスタのタイムアウト**: 動作中は融合演算が活発でストレッチが増え、マスタの読み取りがタイムアウト。BNO055 の既知の癖。
> 2. **U10 の SDA/SCL/VIN/GND ハンダ不良**: 基板のたわみ/振動で瞬断。
> 切り分け: 「U10 近傍を指で押す/基板をたわませる」とエラー増→**ハンダ**。「I2C クロックを下げる」とエラー減→**クロックストレッチ**。
> 校正監視は cal 値が別途更新されるため無害だが、**ナビ（#18）で生 heading を制御に使う前に必須で潰す**。
> **切り分け結果（押下/たわみ試験）**: 本体静止（heading ~229°一定＝融合は静か）で U10 押下/基板たわみを与えると **8%(5/60)** エラー、**無接触静置は 0%(0/46)**。⇒ クロックストレッチ(仮説1)では押下で増えないはずなので否定的、**handling／機械的マージナル(仮説2)が濃厚**。「movement-correlated」は正確には「**handling-correlated（触ると出る）**」。
> 候補箇所（I2C 信号 D14/D15 が通る経路）: ① **Spresense⇔キャリア基板の基板間コネクタ**（Spresense は底面の微細ピッチコネクタで嵌合。BOM に別コネクタ無し）② U10(BNO055モジュール)のハンダ ③ キャリアのトレース。**個別に押して**局所化する（下記プロトコル）。
> **局所化テスト結果（訂正あり）**: 当初 A無接触17% / B U10押下15% / C キャリアたわみ10% / D Spresense押付け0% という単調減少から「基板間コネクタ嵌合不良（押付けで直る）」と推定したが、**追試で逆転・棄却**: 〔押付けたまま静止〕=**12%(7/60、全て手を動かした最初の22サンプルに集中、以降クリーン)** / 〔完全に手放しで静止〕=**0%(0/60)**。**強く押しても改善せずむしろ悪化**＝再嵌合では直らない。
> ⇒ **結論（訂正後）**: 引き金は特定箇所のハンダ/コネクタではなく「**基板を手で触る/動かすこと自体**」。**手を完全に離して静止＝0%、触ると場所に依らず10〜18%**。機構候補: ①人体由来のノイズ結合（容量結合/ESD/EMI。手の動きで過渡的に出て静止で収まる挙動に合致）②手の微振動による接点擾乱（ただし押付けで改善しないので単純な緩みではない）。
> **含意**: 無擾乱なら I2C は健全（0%）。校正時のエラーは「手で持って回す」試験法の副作用が大きい。**ミッションで効くのは走行中の振動＋モータEMI**で、手つつきでは再現不可 → **モータ駆動＋走行振動の別試験（統合フェーズ）が必要**。任意の電気切り分け: 触りながら自分をGND接地でエラー減→人体ノイズ/ESD、I2Cクロック低下/プルアップ強化で減→エッジ健全性。
> ✅ **ソフトの読み取り堅牢化（実装済み, gotchas B11）**: `heading()` を自前 Wire 読み（Euler 0x1A）＋戻り値で成否判定＋リトライ（最大3回, 5ms間隔）にし、失敗時は `0.0` でなく **`kInvalidHeading`** を返す。`imu`/`imu mon` の表示も失敗時は `heading = (invalid: I2C読み取り失敗)`（校正行は継続表示）。値合成はホストテスト済（`compass::eulerHeadingFromRaw`）。走行振動/EMI の瞬断で `0.0=北` を制御に流さない保険。なお Wire コアの `ERROR: Failed to read...` 行はリトライ毎に出るが抑止不可（成否判断は戻り値で行う）。フライトでは Spresense⇔キャリアの機械締結（スタンドオフ＋ネジ）も別途推奨。
> 実装後の実機確認（要再計測）: `imu mon` で①無擾乱時に偽 `0.0` が出ず正常方位が続く ②手で触って瞬断を誘発したとき `(invalid)` 表示になり `0.0` が出ない ③リトライで瞬断頻度が見かけ上減るか、を確認する。なお `ERROR: Failed to read...` 行は**1サンプルの失敗につき最大 kReadAttempts(=3) 行**出る（Wire コアがリトライ毎に printf, 抑止不可）ので、エラー行数から失敗率を出すときは**サンプル単位（`(invalid)` 表示の回数）で数える**こと（行数で数えると最大3倍に膨らむ）。制御ループ（#18）でブロッキングを避けたい場合は単発の `tryReadHeading()` を使い、リトライは周期側で行う。

## 5. 手順D: `isHeadingReady`（暫定しきい値）の妥当性

`isHeadingReady` は `sys==3 && mag==3` で「方位使用可」と判定するが、これは spec 裏付けのない暫定ヒューリスティック（compass.h 参照）。NDOF では gyro も方位ドリフト補正に効くため、**gyro 未校正のまま方位を使うと走り出しでドリフトしうる**。

### 確認

- `imu mon` で **sys=3,mag=3 だが gyro<3** の状態を作り、本体を静止させて heading のふらつき/ドリフトを観察する。
- 同条件で本体をゆっくり一定速度で回し、heading が滑らかに追従するか。

| 条件 | heading のふらつき | ドリフト | 所見 |
|---|---|---|---|
| sys=3,mag=3,gyro<3 |  |  |  |
| 全要素3 |  |  |  |

- [ ] `isHeadingReady`（緩め）で十分 → 採用理由:
- [ ] gyro も要る → 走行開始は `isFullyCalibrated` を既定にする（#18 で適用）
