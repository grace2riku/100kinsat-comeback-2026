# 開発ワークフロー（Issue / ブランチ / PR / レビューゲート）

本リポジトリの開発進行ルール。**TDD 前提・日本語・段階的開発**（CLAUDE.md の方針）に沿う。
狙いは「**Codex の指摘を PR で受ける**」状態から「**PR を出す前にローカルで潰す**」状態へ移すこと（左シフト）。これによりPR上のレビュー往復を構造的に減らし、目的達成まで早く確実に到達する。

---

## 1. Issue 運用

- 作業は必ず Issue 起点。Issue には**目的／完了条件(DoD)／参考**を書く（既存 Issue #2 以降の形式に合わせる）。
- ラベルは `area:*`（領域）と `design`/`test` などを付与。マイルストーン（Phase1〜）に紐付ける。
- 1 Issue = 1 つのまとまった機能。大きすぎる場合は分割する（モジュール単位）。

## 2. ブランチ運用

- main から切る。命名: `feat/issue-<番号>-<短い英語スラグ>`（例: `feat/issue-8-motor-driver`）。
  - 種別接頭辞: 機能=`feat/`、修正=`fix/`、ドキュメント=`docs/`、環境=`chore/`。
- main へ直接 push しない。必ず PR 経由。
- 1 ブランチ = 1 Issue。レビューしやすい粒度を保つ。

## 3. TDD サイクル（実装の基本）

CLAUDE.md の厳守事項に従う。**Red → Green → Refactor**。

1. **Red**: 失敗するテストを先に書く（HW非依存ロジックは `test/` にホストテスト、HW依存は NT-Shell コマンド）。
2. **Green**: テストを通す最小実装。テストを通すためだけのハードコード・`if(testMode)` は**禁止**。
3. **Refactor**: 重複除去・命名整理。テストは緑のまま。
- 正常系だけでなく**境界値・異常系・エラーケース**を必ずテストする（`doc/development/spresense_gotchas.md` B項参照）。

## 4. コミット規約

- 日本語。1コミット1論点。タイトルは「何をしたか」を簡潔に（例: `モータ駆動モジュールをTDDで実装`）。
- 関連 Issue/PR 番号を本文か末尾に含める（`#8` 等）。

---

## 5. PR 前レビューゲート（左シフトの中核）⭐

**push する前に**、ローカルで以下を順に実施する。ここで潰せば PR 上で Codex に指摘される件数が激減する。

### 5-1. 自動チェック（`tools/precheck.sh` 一発）

```bash
tools/precheck.sh
```

これが走らせるもの（CI と同じ内容をローカルで先取り）:
1. **ホストユニットテスト** — `tools/test.sh`（cmake + ctest + doctest）
2. **静的解析** — cppcheck（`src/lib/core`）
3. **整形チェック** — clang-format `--dry-run --Werror`
4. **Arduino ビルド** — `arduino-cli` が在れば対象スケッチを compile（任意）
5. **クロスベンダー別モデルレビュー** — `PRECHECK_CODEX=1` 指定時のみ・任意・人間が手元で実行（§5-3(c)）

全部緑になるまで push しない。これは CI の失敗を**ローカルで先に**観測するための門番。

### 5-2. セルフレビュー チェックリスト（人間／Claude が目で確認）

`doc/development/spresense_gotchas.md` を開き、**今回触った領域**を上から照合する。特に:

- [ ] **A. ターゲット≠ホスト**: 整数の幅(`long`=32bit)・`while(!Serial)`・シリアル幅依存はないか。ホスト緑だけで満足していないか
- [ ] **B. 入力検証**: 外部入力（シリアル/センサ）に `atoi` を使っていないか。範囲チェック・改行(CR/LF/CRLF)・read 戻り値確認はあるか
- [ ] **C. 整合性**: `build.sh`↔`upload.sh`、宣言↔実装↔ドキュメント↔テストの対の更新漏れはないか
- [ ] **D. ドキュメント**: 追加・変更した手順は**実際に操作して動く**か。存在しないUI名・誤った順序はないか
- [ ] **E. ツール前提**: 宣言した最小バージョン（CMake 等）で動くか。生成物の参照先は実在するか
- [ ] **テスト品質**: 意味のないアサーション(`expect(true)`)はないか。境界値・異常系を網羅したか

### 5-3. 独立AIレビュー（PR前の事前批評役）⭐

実装者（メイン会話）とは**独立した視点**で差分をレビューさせ、考慮漏れを push 前に潰す。これがレビュー左シフトの中核。**外部依存ゼロで確実に動く順**に並べる。

**(a) 専用レビューサブエージェント `code-reviewer`（推奨・第一選択）**
- `.claude/agents/code-reviewer.md` に定義済み。本プロジェクトの失敗パターン（`spresense_gotchas.md` の A〜E）を“レンズ”として焼き込んだ敵対的レビュー担当。
- **独立コンテキスト**で動くため、実装者の盲点（ホストtest緑だが32bitターゲットで壊れる 等）を捕まえやすい。
- **外部依存なし**（APIキー・ネット・ネイティブバイナリ不要 ＝ macOS Gatekeeper の影響を受けない）。
- 使い方: Claude Code 上で「`code-reviewer` サブエージェントで差分をレビューして」と指示（または該当タスクで自動委譲）。

**(b) 組込みスキル `/code-review`（汎用バグ観点・第二選択）**
- `/code-review`（effort: low〜max）で差分の正確性バグをレビュー。`/code-review ultra` は多エージェントのクラウドレビュー。
- `--comment` でPRにインラインコメント、`--fix` で修正適用も可能。
- (a) がプロジェクト特化、(b) が汎用。両方かけると網羅性が上がる。

**(c) クロスベンダー第2モデル（任意・多様性の確保）**
- **GitHub の Codex 自動レビュー**は PR で自動的に走る（OpenAI公式・稼働中）。別ベンダーの目として有効。
- さらに push 前に別モデル（OpenAI/Gemini 等）へ通したい場合は、HTTPS API 経由（フックの `http`/`mcp_tool` または MCP サーバ）で接続する。APIキー・コストが必要で、Anthropic 公式の事例ではない点に留意。
- ローカル `codex` CLI を使う場合は、macOS Gatekeeper の許可（`システム設定→プライバシーとセキュリティ→このまま開く`）が前提。**Claude Code 自身は codex を起動できない**（サンドボックス/Gatekeeper のため）ので、これは人間が手元で回すステップ。

> どの手段でも、新しい指摘を受けたら `spresense_gotchas.md` に1行追加して資産化する（指摘を恒久ルールに変えるのが本仕組みの目的）。

## 6. PR 運用

- タイトル: 日本語で「何をしたか (#Issue番号)」。本文に**目的・変更点・テスト結果・関連Issue (`Closes #N`)** を書く。
- PR 前レビューゲート（§5）を通してから出す。
- Codex の自動レビューが付く。**妥当な指摘は必ず対応**し、対応コミットと「何が原因でどう直したか」を返信する（過去PRの良い慣行を踏襲）。
- CI（host-test / lint / arduino-build）が全て緑であること。
- main へは Squash & Merge。マージ後ブランチは削除。

## 7. Claude Code / Codex の役割分担

| 局面 | 担当 | 補足 |
|---|---|---|
| 実装（TDD） | **Claude（メイン会話）** | Red→Green→Refactor。`gotchas` を実装中に参照 |
| PR前セルフレビュー | **Claude（メイン会話）** | §5-2 チェックリスト |
| PR前 独立レビュー（中核） | **`code-reviewer` サブエージェント** | §5-3(a)。独立コンテキスト・外部依存ゼロ・確実に動く |
| PR前 汎用バグレビュー | **`/code-review` スキル** | §5-3(b)。`ultra` で多エージェント |
| PR 自動レビュー（クロスベンダー） | Codex（GitHub） | §5-3(c)。別ベンダーの目。残った指摘の最終網 |
| 知見の蓄積 | 全担当 | 指摘は都度 `gotchas` に1行追加 |

中核は「実装は Claude の強み（TDD・仕様整合）を活かしつつ、**独立コンテキストのサブエージェント**を事前批評役に回す」こと。これは外部依存ゼロで毎回確実に動く。クロスベンダーの多様性は GitHub Codex（および任意の第2モデルAPI）で補う。PR 上の往復を最小化するのが目的。
